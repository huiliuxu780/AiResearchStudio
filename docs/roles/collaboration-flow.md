# 多角色协作流程（PM / 前端 / 测试）

## 目标
- 在不扩大范围的前提下，稳定推进 research workbench 开发与发布。

## 流程
1. PM 创建需求单并标注优先级与验收标准。
2. 前端评审可实现性，确认组件复用方案（shadcn-first）。
3. PM 锁定本轮范围，进入开发。
4. 前端完成实现并提交自测记录。
5. 测试执行冒烟和增量回归，输出缺陷清单与分级。
6. PM 组织 Go/No-Go，满足门禁后发布。

## 会议节奏
- 每日站会：进度、阻塞、风险。
- 周中联调：按验收项逐条过。
- 发布前评审：质量签字与风险确认。

## 升级规则
- 涉及 P0/P1 缺陷时，立即升级到当日优先处理。
- 发布冻结期仅允许 P0 变更进入。
- 所有临时变更必须补齐变更单与回滚方案。

## 收工交接流程（新增）
- 触发条件：用户明确说“今天先到这里/结束今天开发/收工”。
- 必做动作：
  1. 前端执行最小回归（type-check、lint）。
  2. 测试确认当前已知风险和待验证项。
  3. PM 确认本轮范围边界，避免未完成项被误判为已完成。
  4. 主 Agent 统一提交并推送远端。
  5. 主 Agent 输出“下次启动开发提示词”。
- 交接产物：
  - 最新 commit hash
  - 已完成清单
  - 未完成清单
  - 风险/阻塞
  - 下一步第一任务
